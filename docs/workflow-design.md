# 工作流模块技术方案

## 核心思路：自然语言 → 工作流

用户不需要拖拽连线，只需要用文字描述需求，后台 LLM 自动解析成结构化工作流。

---

## 数据模型

复用 `ai_flow` 表：

| 字段 | 用途 |
|------|------|
| `name` | LLM 根据描述生成的工作流名称 |
| `description` | 用户的原始自然语言描述 |
| `chain` | LLM 生成的步骤列表（人可读，如 "定时触发 → 检索素材库 → 写笔记"） |
| `design` | 结构化 JSON（机器可执行，含 nodes / edges） |
| `metadata` | 额外配置（如 `{"schedule": "0 8 * * *"}`） |

## 技术选型说明

### 图片生成 — 为什么用 `OpenAiImageModel`
LangChain4j 已封装，通义千问/DeepSeek 都兼容 OpenAI 格式，换供应商只改 URL 和模型名。

### 条件判断 — 用 LLM 做路由，但不硬编码分类
- 分类来自用户工作流描述，不预设候选列表
- 用户在描述里写"美食类用温柔风，科技类用专业风"，LLM 解析时提取分支条件
- 运行时把分类标准和内容一起交给 LLM，让 LLM 自由判断走哪条路
- 没有匹配的分支就走默认路径
- 这样才真正灵活，不是换个方式硬编码

实现方式：
```
// ❌ 错误：硬编码分类
String result = llm.chat("属于旅游/美食/科技？");
if ("旅游".equals(result)) ...

// ✅ 正确：分类来自工作流描述，运行时 LLM 自由判断
String branches = workflow.getBranches(); // 用户描述里提取的
String result = llm.chat("根据以下分类标准判断内容走哪条路：" + branches);
slot.setChainId(result); // 没有匹配就走默认
```

### 发送/发布 — 为什么用 OAuth2
用户授权一次，后端存 token，后续自动发布，不需要每次手动登录。

### 定时触发 — 为什么用 `@Scheduled`
Spring Boot 内置，一行注解即可。博主场景每天跑一次/几小时一次，够用。

### 数据抓取 — 为什么用 Brave Search
已有集成，直接复用。合法合规的搜索 API，非爬虫。
