# 工作流模块技术方案

## 核心思路：自然语言 → 工作流

用户不需要拖拽连线，只需要用文字描述需求，后台 LLM 自动解析成结构化工作流。

---

## 数据模型

复用 `ai_flow` 表：

| 字段 | 用途 |
|------|------|
| `name` | LLM 根据描述生成的工作流名称 |
| `description` | 用户的原始自然语言描述 |
| `chain` | LLM 生成的步骤列表（人可读，如 "定时触发 → 检索素材库 → 写笔记"） |
| `design` | 结构化 JSON（机器可执行，含 nodes / edges） |
| `metadata` | 额外配置（如 `{"schedule": "0 8 * * *"}`） |

---

## API 设计

| 方法 | 路径 | 说明 | 状态 |
|------|------|------|------|
| POST | `/api/ai/flow/generate` | 接收文本，LLM 解析成工作流并存库 | 新增 |
| GET | `/api/ai/flow/list` | 列出所有工作流 | 已有 |
| POST | `/api/ai/flow/{id}/run` | 执行工作流 | 已有 |
| PUT | `/api/ai/flow/{id}` | 更新工作流 | 已有 |
| DELETE | `/api/ai/flow/{id}` | 删除 | 已有 |

## 技术选型说明

### 图片生成 — 为什么用 `OpenAiImageModel`
LangChain4j 已封装，通义千问/DeepSeek 都兼容 OpenAI 格式，换供应商只改 URL 和模型名。

### 条件判断 — 功能路由，不是内容分类
条件分支的作用是**决定下一步调用哪个功能节点**，不是给内容打标签。

正确场景：
```
LLM 判断用户意图
    ├─ "查资料" → 走知识库检索节点
    ├─ "写文案" → 走 LLM 生成节点
    └─ "找图片" → 走图片生成节点
```

或：
```
知识库检索
    ├─ 找到结果(相似度>0.75) → 走 LLM 生成（基于资料回答）
    └─ 没找到结果(相似度<0.75) → 走联网搜索兜底
```

### 发送/发布 — 为什么用 OAuth2
用户授权一次，后端存 token，后续自动发布，不需要每次手动登录。

## 可观测性要求

### 联网搜索
- 每条搜索结果必须附带来源链接，前端展示为可点击的 URL
- 返回格式：
```json
{
  "title": "标题",
  "url": "https://...",
  "snippet": "摘要"
}
```

### 知识库检索
- 每个检索结果必须标注来源：文档名 + 第几段（chunk_index）
- 用户可以看到"这段话来自《XX.docx》第3段"
- 返回格式：
```json
{
  "content": "匹配内容",
  "docName": "文档名称",
  "chunkIndex": 3,
  "similarity": 0.89
}
```

### 通用原则
- 前端所有展示给用户的信息，都必须标明来源
- LLM 生成的内容标注"AI 生成"，检索到的内容标注来源文档
- 用户能追溯到每一条信息的出处

### 定时触发 — 为什么用 `@Scheduled`
Spring Boot 内置，一行注解即可。博主场景每天跑一次/几小时一次，够用。

### 数据抓取 — 为什么用 Brave Search
已有集成，直接复用。合法合规的搜索 API，非爬虫。
